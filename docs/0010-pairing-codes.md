## Decision 10
Question
--------
How should a second (or Nth) device join an account without always typing the
account password?

Options
-------
A. Password on every new device (Phase 1 status quo only)
B. Short-lived pairing code generated by an already-linked device
C. Magic link / email OTP
D. QR code encoding a one-time secret (UI-heavy)

Decision
--------
A remains available for bootstrap and recovery. B is added for normal
multi-device linking.

Flow:
1. Linked device (authenticated) calls `POST /v1/pairing-codes` → server
   returns an 8-character crockford-base32 code, TTL ~10 minutes, single-use.
2. New device calls `POST /v1/pairing-codes/redeem` with the code + device
   name/platform → server creates device + token + keys (same as login link)
   and consumes the code.
3. Codes are stored hashed (SHA-256), like bearer tokens.
4. Optional: creator can cancel an unused code.

Email OTP and QR are deferred (need mailer / UI). Password login stays for
"no other device online" and first account creation.

Reason
------
Typing the account password onto every phone is a phishing and shoulder-surf
risk, and it trains users to enter the master secret often. A short code from
a device you already trust matches how Syncthing-style and many 2FA-adjacent
flows work, needs no email infrastructure, and fits a CLI agent (`pair`
command). QR is just a transport for the same code later.
